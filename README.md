## HDD와 SDD의 장단점
- HDD
> 장점 : 
> 1. 용량당 가격이 저렴(TB단위에서는 독무대
> 1. 데이터 장기보존이 가능(SSD의 경우 방전으로 데이터 유실 가능성 존재)
> 단점 : 
> 1. 소음 및 전력 문제, 발열
> 1. 진동이나 충격의 영향을 많이 받음
> 1. 느린 속도

- SSD
> 장점 : 
> 1. 빠른 속도(플레시 메모리를 사용하여 빠른 속도를 보임)
> 1. 견고함(구동 부분이 없어 소리가 나지 않고 진동이나 충격에 영향을 받지 않음)

> 단점 : 
> 1. HDD대비 비싼 가격과 적은 용량(용량대비 높은 단가)
> 1. 덮어쓰기 횟수의 제한(짧은 수명)


## Git을 왜 사용해야 할까? 실제 사용에서 Git과 유사한 경험을 한적은?
- 팀작업시에 누가 무엇을 변경했는지 기록이 남기 때문에 팀원간 공동작업에 도움이 됨 - 공동작업시 어떤 부분이 수정되었는지 확인이 필요한 부분이 있고, 신규 업데이트 버젼이 오류가 있을 경우 전 버젼과의 비교가 필요하기 때문에 변경 기록을 계속해서 저장하는 것이 공동작업을 하는 데 용이함 - 구글드라이브를 활용해서 문서작업을 공유하거나 마이크로 소프트 팀즈를 통해 파일공유와 파일 수정등을 동시해 진해본 적 있음



## Git과 Github의 차이점
- Git : 변경사항 추적 시스템
> 파일의 변화기록을 전부 저장해 놓을 수 있어, 버젼관리에 용이
> 이전 버젼으로 돌아가거나 다른 사람들이 변경 기록을 쉽게 확인 할 수 있음
- GitHub : 변경사항을 저장할 수 있는 클라우드 서비스
> Git을 클라우드에 저장해 데이터 유실에 대비할 수 있고, 비주얼적으로 볼 수 있다는 장점

## 인터넷은 어떻게 동작할까?
- 컴퓨터가 서버와 통신하는 과정
> 인터넷 주소(ex. Google.com)를 입력하면 DNS를 통해 IP규칙에 따라 주소가 IP형태(ex. 211.39.987.01)로 바뀌고, 패킷 단위로 나눠져서 네트워크에 도달.
> 브로드캐스트를 통해 물리적 주소를 받아와서 해당하는 랜카드가 있는 컴퓨터에 접근
> 우리가 원하는 데이터를 가져오게 됨
- IPv4 와 IPv6을 사용하는 이유와 특징

| 구분 | IPv4 | IPv6 |
| ------- | ---------- | ---------- |
| 주소길이 | 32비트 | 128비트 |
| 주소 개수 | 약 43억개 | 거의 무한대 |
| 품질제어 | 품질 보장 곤란 | 등급별, 서비스 별로 패킷을 구분할 수 있어 품질 보장 용이 |
| 보안 기능 | IPsec프로토콜 별도 설치 | 확장 기능에서 기본으로 제공 |
| 자동 네트워킹 | 곤란 | 있음 |
| 이동성 지원 | 곤란(비효율적) | 용이(효율적) |

- GET과 POST의 3가지 차이점
> 1) GET은 주소줄에 값이 ?뒤에 쌍으로 이어붙고 POST는 숨겨져서(body안에) 보내짐
> 2) GET은 URL에 이어붙기 때문에 길이제한이 있어서 많은양의 데이터는 보내기 어렵고 POST는 많은 양의 보내기에도 적합(용량제한은 있지만)
> 3) 즉 http://url/bbslist.html?id=5&pagenum=2 같이 하는 것이 GET방식이고 form을 이용해서 submit을 하는 형태가 POST

## 앱디자인 체크리스트
### 디자인을 하는 순서
- Brainstorm >  Plan > Prototype > Evaluate
- 선정, 조사, 정의, 정리, 화면디자인, 검토, 프로토타이핑화, 확인

### 앱화면을 디자인 화면으로 인식하면 안됨
- 앱은 반드시 인터렉션이 발생
- 앱의 인터렉션은 사용자의 손에 의해 동작
- Apple의 HIG 반드시 이해필요

### 사용자의 환경을 이해
- 화면 배치와 사용성 배치
- 선택이 유라하게 되어 있는지 검토하기(한손 엄지로 누르기 쉽게)
- 화면이 커질수록 하단 사용이 주요해짐
- 첫 화면에 보이지 않는 컨텐츠들
> 사람들이 스크롤을 하지 않는다는 고정관념이 있지만 실제로는 스크롤을 한다(단, 내용이 흥미롭거나 context 있는 내용들일 경우 / 지루하거나 지칠 때 스크롤링을 멈춤
> 반대로, 숨은 UI처럼 예상이 되지 않은 연관성 없는 컨텐츠를 보기 위해 좀처럼 스크롤을 하지는 않을 수 있음

### 앱이 ADS를 만족하고 있는가?
- 컨셉의 일관성 유지
- 적어도 최초의 목적을 달성할 수 있음
- 팀원 간의 공동 목적 설정
- 진행하는 내내 지속적으로 확인필요
- 목표 완수 후, 테스트와 피드백에 따라 그 궤도가 수정될 수는 있음

### 앱의 흐름
- 가장 중요한 컨텐츠를 중심으로
- 사용자가 각 탭을 통해 그들이 얻어낼 내용을 유추할 수 있어야 함
- 앱 내에는 길치가 없어야 함
> 2nd depth초과하여 진입하는 네비게이션은 다른 방안을 재검토
> 되돌아가는데 부담은 다시는 들어가고 싶지 않음
> tab bar 항시 사용할 수 있도록 활용하자
- 실행 취소가 용이한가?
> 뭔가를 선택하지 않고 이전으로 되돌아가는 법, 현 상황에서 나가는 법에 방법을 쉽게 인지할 수 있게 해야함(고민없이 즉각 수행할 수 있게)
> 닫기 혹은 취소 버튼 활용

### 명확한 Task를 수행하는 가?
- 한화면에서는 그 화면에 해당하는 1개의 Task
> 사용자의 빠른 판단과 사용 이해도 떨어짐
>> 앱을 통해 무엇을 얻게 될지 판단 시간이 소요되거나
>> 복잡한 환경을 즉각 벗어나게 될 수도 있음
> 화면의 기능이 즉각적으로 이해되게 명확한 전달 필요
> 여러 기능을 한번에 담는 홈화면 같은 상황을 만들지 말자
- Task에 부합하는 명확한 기능 제공
- 화면과 연관성이 떨어지면  그 기능을 찾을 확률이 떨어짐

### 기본 UI를 충실하게 사용하는가?
- 즉각적인 컨트롤이 가능한 UI 우선 사용
> 입력값을 선택할 때마다 modal window나 bottom sheet를 이용하는 방법은 가장 최후 고려 사항
- 동시 컨트롤 활성화 X
> 사용자가 조작하고 있는 것을 알 수 있게 해야함
>> 현재 무엇을 하고 있는지 알 수 없게 될 수 있음
- 명확한 터치 가능성

### Tab Bar 
- 최상위 컨텐츠를 최단 시간의 이동할 수 있기 때문에 주로 사용됨(가장 큰 골조)
- Tab bar 규칙 (HIG)
> 5개 이하 탭(HIG), 3~5개 탭(Material design), (최소 단위 44point 이상 권장)
> 5개를 초과할 경우, 5번째 탭에 ‘더보기’식 메뉴 활용
> Tool bar와 동시 사용 X (하단 영역이 겹쳐지기 때문)
- HIG 최소 44 point 크기의 컨트롤 제공
- Tab bar 팁
> 탭에 아이콘만 있다면 인지성이 떨어짐
>> 텍스트 동시 사용 권장
>> 탭 아이콘은 픽토그램 수준의 인식률 필요
>> 많은 컬러 X (단일 컬러 O)
> 탭과 탭 간은 즉시 이동 (모션 X)
>> 빈번히 선택되는 탭(버튼)은 눈요기 거리의 모션은 어울리지 않음
>> 안드로이드의 상단 탭에서 사용하는 횡 방향의 슬라이드 동작의 맥락을 갖지 않음
> 좌부터 1번, 2번, 3번 탭 순으로 사용률이 나타남
>> 해당 탭의 첫 화면에서 사용률이 높게 나타남
- Tab Bar와 버튼 (ex. 인스타의 사진 추가 버튼)
> 탭바로 오인 가능하지만 인스타의 경우 손이 잘 가는 위치에 버튼을 넣은 것으로 생각할 수 있음
> Tab bar 중앙에 버튼(사용성이 올라감)

### 숨은 UI, Tab bar와 Slide menu(햄버거 메뉴)
- 햄버거 메뉴를 통한 진입 확률은 1/3 (닫혀 있는 메뉴에 의도적으로 접근하는 사용자 적음)
- 2016년 3월 Material design 가이드에 Bottom navigation(하단 탭바) 추가
- 항상 노출되는 Tab bar 사용률 증가
> 고객이 앱을 더 자주 이용( 세션수 2배 이상 증가)
> 앱에서 더 많은 시간 체류
- 메뉴 수가 적을 수록 사용자는 편리해짐
> 햄버거메뉴를 통한 다양한 기능보다 탭바를 이용한 중요 컨텐츠 노출이 사용성을 더 높여줌
- 햄버거 메뉴는 2nd depth진입시 백버튼의 위치가 애매함

### 사용자의 선택 시간을 줄여주는 UI를 고려해야 함
- 시각적 고려 후 반드시 사용성을 고려해야 함
- 지그재그 시선은 구성요소 이해의 속도와 사용속도를 저하시킴

### 극적인 상황을 고려해야 함
- 가장 간단하게 인식하고,
- 가장 간단하게, 빠르게 사용할 수 있는가
- 구성 요소를 파악할 필요가 없을 정도로
- 직관적이게 수성
- 초등학생 또는 고령의 사용자도 인식이 가능하게
- 실외에서는 미묘한 색구분이 어려울 수 있음(색구분을 명확하게)
- 바로 가입하고 결제하기 보다는 체험 후 인앱 결제 할 수 있게
